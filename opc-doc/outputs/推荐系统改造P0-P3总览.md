## P0-1：清除调试残留代码

### 改造前

`RecommendationService.java` 第 528-535 行硬编码了 4 只调试股票：

```java
// 调试代码残留
List<String> debugStocks = Arrays.asList("002033", "601000", "600258", "601008");
candidates = candidates.stream()
    .filter(c -> debugStocks.contains(c.getCode()))
    .collect(Collectors.toList());
```

**问题**：生产环境每次推荐只从这 4 只股票中选，完全绕过了因子打分和策略筛选流程。推荐结果不可信。

### 改造后

删除 L528-535 调试代码块，推荐流程恢复正常的因子打分 → 策略筛选 → 置信度排序链路。

### 改造价值

| 维度 | 说明 |
|------|------|
| 正确性 | 推荐结果恢复全市场筛选，不再被调试代码劫持 |
| 安全性 | 防止调试代码意外影响生产数据 |
| 可信度 | 推荐结果可以用于实盘参考 |

### 验证方式

```sql
-- 查询最近推荐是否覆盖多只股票（不再只有4只）
SELECT strategy_id, COUNT(DISTINCT symbol) AS stock_count
FROM stock_recommendation
WHERE recommend_date >= CURDATE() - INTERVAL 7 DAY
GROUP BY strategy_id;
-- 预期：每个策略推荐股票数 > 4
```

## P0-2：超额收益追踪

### 改造前

`trackRecommendationPerformance()` 只追踪推荐股票的**绝对收益**（次日收盘价 / 推荐日收盘价 - 1）。

**问题**：
- 大盘涨 3%，推荐股票涨 1% → 绝对收益为正，但实际跑输大盘
- 置信度评分的"命中率""平均收益""回撤""波动率"全部基于绝对收益，无法区分"选股能力"和"市场 beta"
- 牛市中所有策略置信度虚高，熊市中虚低

### 改造后

`StockRecommendation` 实体新增 3 个字段：

| 字段 | 含义 |
|------|------|
| `nextDayExcessReturn` | 次日超额收益 = 个股收益 - 沪深300收益 |
| `nextWeekExcessReturn` | 次周超额收益 |
| `nextMonthExcessReturn` | 次月超额收益 |

`calculateDimensions()` 优先使用超额收益计算命中率/平均收益/回撤/波动率，旧数据（无超额收益字段）回退到绝对收益。

**DDL**：`add_excess_return_columns.sql`

**存储单位约定**：百分比值（如 -7.03 表示 -7.03%），非小数。显示时不再 ×100。

### 改造价值

| 维度 | 说明 |
|------|------|
| 评分准确性 | 置信度反映"选股能力"而非"市场涨跌"，牛熊周期中性 |
| 策略评估 | 可区分 alpha 策略和 beta 策略，淘汰伪 alpha |
| 归因分析 | 超额收益 + 绝对收益双维度，支持 Brinson 归因 |

### 验证方式

```sql
-- 1. 确认字段存在且有数据
SELECT symbol, recommend_date,
       next_day_return, next_day_excess_return
FROM stock_recommendation
WHERE next_day_excess_return IS NOT NULL
ORDER BY recommend_date DESC LIMIT 10;
-- 预期：excess_return ≈ next_day_return - 沪深300当日涨幅

-- 2. 超额收益应正负都有（非全正=真实选股而非市场涨跌）
SELECT
  SUM(CASE WHEN next_day_excess_return > 0 THEN 1 ELSE 0 END) AS positive,
  SUM(CASE WHEN next_day_excess_return <= 0 THEN 1 ELSE 0 END) AS negative
FROM stock_recommendation
WHERE next_day_excess_return IS NOT NULL;
```

## P0-3：冷启动加速

### 改造前

`StrategyConfidenceService` 中，策略历史推荐样本 < 20 期时返回 `UNTRAINED` 状态，不参与置信度评分，推荐数 = topN（无缩减）。

**问题**：
- 新策略上线后 20 天内（每天 1 期）置信度始终 UNTRAINED
- topN 不缩减 = 全量输出，推荐质量无保障
- UNTRAINED 状态下无法进入评分系统，冷启动期太长

### 改造后

新增 `MIN_SAMPLES_FOR_SCORING = 5` 和 `createColdStart()` 方法：

| 样本数 | 状态 | 行为 |
|--------|------|------|
| < 5 | COLD_START | 贝叶斯收缩：先验 60% + 实际 40%，先验分 = 55（NORMAL 下限） |
| 5 ~ 19 | WARMING_UP | 正常评分但缩减 topN |
| >= 20 | TRAINED | 完整置信度评分 |

冷启动不缩减 topN 但进入评分系统，比 UNTRAINED 更快有输出。

### 改造价值

| 维度 | 说明 |
|------|------|
| 冷启动速度 | 从 20 天缩短到 5 天即有评分输出 |
| 安全性 | 先验分 55 = NORMAL 下限，保守不冒进 |
| 渐进性 | 贝叶斯收缩平滑过渡，避免样本少时评分剧烈波动 |

### 验证方式

```sql
-- 1. 查询策略置信度状态分布
SELECT cs.strategy_id, s.strategy_name, cs.status, cs.sample_count
FROM strategy_confidence cs
JOIN strategy s ON cs.strategy_id = s.id;
-- 预期：新策略 status = COLD_START 或 WARMING_UP，非 UNTRAINED

-- 2. 手动触发推荐，验证冷启动策略有置信度评分
-- API: POST /api/recommendation/generate?dateMode=today
-- 预期：冷启动策略推荐带 confidence_score（非 null）
```

## P1-4：噪声因子剔除

### 改造前

推荐管线 P1（IC 计算阶段）中，因子 IC 在 [-0.015, +0.015] 区间时执行**反转**（IC < 0 → 取绝对值加权），认为负 IC 是反向指标。

**问题**：
- |IC| < 0.015 的因子预测能力极弱，统计上不显著（噪声）
- 反转噪声因子 = 给噪声赋予方向性，放大随机波动
- 低 IC 因子反转后偶尔"碰巧"表现好，干扰权重分配

### 改造后

新增 `NOISE_FACTOR_IC_THRESHOLD = 0.015`：

| IC 绝对值 | 处理方式 | 状态 |
|-----------|----------|------|
| < 0.015 | 直接剔除 | `NOISE_DROPPED` |
| 0.015 ~ 0.03 | 反转（IC < 0） | `REVERSED` |
| > 0.03 | 正常保留 | `KEPT` |

在 KEPT 分支前检查噪声阈值，`NOISE_DROPPED` 因子不参与后续 ICW 权重计算。

### 改造价值

| 维度 | 说明 |
|------|------|
| 信噪比 | 剔除统计不显著的因子，ICW 权重集中在真正有效的因子上 |
| 稳定性 | 避免噪声因子反转后随机波动干扰权重分配 |
| 可解释性 | `NOISE_DROPPED` 状态明确标记被剔除原因 |

### 验证方式

```sql
-- 1. 查询因子 IC 管线处理结果
SELECT factor_code, ic_value, ic_status
FROM factor_ic_log
WHERE calc_date = CURDATE();
-- 预期：|ic_value| < 0.015 的因子 ic_status = 'NOISE_DROPPED'

-- 2. 确认 NOISE_DROPPED 因子不在 ICW 权重中
SELECT factor_code, icw_weight
FROM factor_icw_result
WHERE calc_date = CURDATE();
-- 预期：不含 ic_status = 'NOISE_DROPPED' 的因子
```

## P1-5：PATTERN 策略接入 IC 管线

### 改造前

PATTERN 策略（形态策略）的 `PatternDetector` 输出 0-100 的连续评分，但这个信号**不参与**因子 ICW 权重计算。

**问题**：
- 形态信号只在策略层使用，无法通过 IC 机制验证其预测能力
- 形态信号无法与其他因子（SIZE、FIN_ROE 等）在同一权重框架下比较
- ICW 候选池只有 BUILTIN 因子，PATTERN 信号被排除在外

### 改造后

**1. 注册 PATTERN 伪因子**
- `FactorDefinition.FactorType` 枚举新增 `PATTERN`（原 BUILTIN/SCRIPTED/COMPOSITE）
- 新增 6 个 PATTERN 伪因子计算器（PatternStrengthCalculator 等），输出 0-1 归一化信号
- `factor_definition` 表插入 6 条 PATTERN 类型因子定义

**2. 修复 FactorComputeEngine**
- `computeSingleValue()` 原来只检查 `FactorType.BUILTIN`，PATTERN 类型返回 null → 0 行写入 CH
- 修复：添加 `|| FactorType.PATTERN` 条件，PATTERN 也走 `builtinCalculators` 查找路径

**3. ICW 默认权重调整**
- 无 IC 历史数据时默认权重从 1.0 → 0.05
- 防止新因子（如 PATTERN_STRENGTH）因无 IC 数据占据 87% 权重

**4. 策略配置**
- 策略 73/76 的 `factorConfigJson` 添加 `PATTERN_STRENGTH` 因子

### 改造价值

| 维度 | 说明 |
|------|------|
| 因子统一 | 形态信号进入 ICW 框架，与基本面/估值因子统一比较 |
| IC 验证 | 形态信号可通过 IC 验证预测能力，淘汰无效形态 |
| 权重安全 | 默认权重 0.05 防止新因子冷启动期权重过高 |

### 验证方式

```sql
-- 1. 确认 PATTERN 因子已注册
SELECT factor_code, factor_type, status
FROM factor_definition
WHERE factor_type = 'PATTERN';
-- 预期：6 条记录，status = 'ACTIVE'

-- 2. 确认 PATTERN 因子有数据写入 CH（修复 FactorComputeEngine 后）
-- ClickHouse 查询：
-- SELECT factor_code, count() AS rows
-- FROM factor_value
-- WHERE factor_code LIKE 'PATTERN_%'
--   AND calc_date = today()
-- GROUP BY factor_code;
-- 预期：6 个因子各约 5000+ 行（全市场覆盖）
-- 修复后实际：6 个因子共 765 万行

-- 3. 确认 PATTERN 因子进入 ICW 权重
SELECT factor_code, icw_weight
FROM factor_icw_result

  AND factor_code LIKE 'PATTERN_%';
-- 预期：权重 ≈ 0.05（冷启动默认值）
```

## P1-6：个股分析并行化

### 改造前

个股深度分析（`AnalysisService.analyzeStock()`）使用串行 `for` 循环逐只处理：

```java
for (String symbol : symbols) {
    AnalysisResult result = analyzeStock(symbol);  // 每只 2-5 秒
}
```

**问题**：
- 50 只股票 × 3 秒/只 = 150 秒，前端超时
- CPU 多核闲置，IO 等待期间无并行收益
- 用户体验差，等待时间长

### 改造后

使用 `CompletableFuture` + 5 线程池并行：

```java
List<CompletableFuture<AnalysisResult>> futures = symbols.stream()
    .map(s -> CompletableFuture.supplyAsync(() -> analyzeStock(s), executor))
    .collect(Collectors.toList());
CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
    .get(60, TimeUnit.SECONDS);  // 60秒超时
```

### 改造价值

| 维度 | 说明 |
|------|------|
| 性能 | 50 只股票从 150s → 30s（5x 并行），约 5 倍加速 |
| 用户体验 | 30s 可接受，不再前端超时 |
| 可扩展性 | 线程池可调，未来加机器可直接提升并发数 |

### 验证方式

```bash
# API 调用计时
time curl -X POST "http://localhost:8080/api/analysis/batch" \
  -H "Content-Type: application/json" \
  -d '{"symbols": ["600519","000858",...共50只],"dateMode":"today"}'
# 预期：响应时间 < 60s（原 > 120s）
```

## P2-7：Regime 权重回验

### 改造前

Regime 权重（BULL/BEAR/SIDEWAYS）使用硬编码值（如 0.6/0.4/0.5），无历史回验验证这些权重是否最优。

**问题**：
- 硬编码权重可能不适应当前市场环境
- 无法量化"当前权重 vs 其他权重方案"的表现差异
- 缺乏权重调整的数据支撑，调参靠直觉

### 改造后

新增 `RegimeBacktestService`：

**1. 历史 Regime 分布统计**
- 统计过去 N 天（默认 120 天）的 Regime 分布占比

**2. 按 Regime 分组推荐表现**
- 对每个 Regime，计算推荐胜率、平均超额收益、样本量
- 回答"在 BULL 环境下推荐表现如何？"

**3. 权重最优性验证**
- 对比 4 种权重方案：当前权重 / 激进权重 / 保守权重 / 静态权重
- 计算每种方案的预期超额收益
- 输出权重调整建议

**API**：`GET /api/regime-backtest/run?days=120`

### 改造价值

| 维度 | 说明 |
|------|------|
| 数据驱动 | 权重调整有历史数据支撑，不再靠直觉 |
| 适应性 | 可定期回验，适应市场环境变化 |
| 可解释性 | 输出"为什么建议调整权重"的具体依据 |

### 验证方式

```bash
# 调用 Regime 回验 API
curl "http://localhost:8080/api/regime-backtest/run?days=120"
# 预期输出：
# - Regime 分布（如 SIDEWAYS 91.7%）
# - 各 Regime 胜率（如 SIDEWAYS 54.7%）
# - 权重最优性对比表
# - 权重调整建议
```

## P2-8：行业相关性动态化

### 改造前

`RecommendationService` 中 `INDUSTRY_CORR_GROUPS` 是 8 组**硬编码**的静态行业相关性分组：

```java
private static final Map<String, List<String>> INDUSTRY_CORR_GROUPS = Map.of(
    "金融", List.of("银行", "证券", "保险"),
    "消费", List.of("食品饮料", "商贸零售", "纺织服装"),
    // ... 8 组硬编码
);
```

**问题**：
- 行业相关性随市场环境变化（牛熊周期、政策变化），硬编码无法适应
- 静态分组可能过时（如新能源行业崛起后与传统能源相关性变化）
- 无法反映短期事件驱动的相关性变化

### 改造后

新增 `DynamicIndustryCorrelationService`：

**1. 动态计算**
- 基于申万一级行业指数近 60 日收益率计算相关系数矩阵
- 使用 Union-Find 聚类算法（阈值 0.70）自动分组

**2. 缓存机制**
- 6 小时缓存 TTL，避免每次推荐都重新计算
- 计算失败时回退到静态 `INDUSTRY_CORR_GROUPS`

**3. 集成**
- `RecommendationService.getCorrGroup()` 优先使用动态分组
- 静态分组保留作 fallback

### 改造价值

| 维度 | 说明 |
|------|------|
| 适应性 | 行业相关性随市场自动更新，不再依赖人工维护 |
| 准确性 | 60 日滚动窗口反映近期相关性，比静态分组更准确 |
| 鲁棒性 | 动态计算失败时回退静态分组，不阻断推荐流程 |

### 验证方式

```bash
# 1. 后端启动日志确认动态计算成功
grep "DynamicIndustryCorrelation" backend/logs/quant-platform.log | tail -5
# 预期：日志显示聚类结果（如 "动态行业分组：金融=[银行,证券,保险], ..."）

# 2. 触发推荐，验证使用了动态分组
curl -X POST "http://localhost:8080/api/recommendation/generate?dateMode=today"
# 检查日志中是否出现 "使用动态行业分组" 而非 "使用静态行业分组"
```

## P2-9：策略组合层风控

### 改造前

多个策略各自独立推荐，推荐结果直接合并，无跨策略风控检查。

**问题**：
- 同一只股票可能被 3 个策略同时推荐 → 隐性仓位超标
- 同一行业被多策略推荐 → 行业集中度过高
- 组合级回撤无人监控 → 极端行情下无预警
- 多策略推荐合并后无去重逻辑

### 改造后

新增 `PortfolioRiskService`：

| 风控维度 | 规则 | 上限 |
|----------|------|------|
| 个股去重 | 跨策略推荐同一股票，取置信度最高的一条（BEST_ONLY 模式） | — |
| 行业集中度 | 合并后组合中单一行业占比 | ≤ 35% |
| 单股仓位 | 跨策略合并后单只股票总权重 | ≤ 15% |
| 组合回撤 | 基于模拟盘 paper_nav 历史峰值计算回撤 | ≤ 20% |

**集成**：`DAILY_RECOMMENDATION` 完成后自动执行组合风控检查（多策略时）。

**API**：
- `GET /api/portfolio-risk/check` — 全量风控检查
- `GET /api/portfolio-risk/drawdown` — 回撤报告

### 改造价值

| 维度 | 说明 |
|------|------|
| 仓位安全 | 防止隐性仓位超标，单股不超过 15% |
| 分散度 | 行业集中度 ≤ 35%，避免行业风险暴露 |
| 预警 | 组合回撤 > 20% 时预警，极端行情下保护 |
| 自动化 | 每日推荐后自动执行，无需人工干预 |

### 验证方式

```bash
# 1. 全量风控检查
curl "http://localhost:8080/api/portfolio-risk/check"
# 预期：返回 50 条推荐的风控检查结果（通过）

# 2. 回撤报告
curl "http://localhost:8080/api/portfolio-risk/drawdown"
# 预期：返回模拟盘组合回撤（如 0.61%，远低于 20% 上限）

# 3. 查看每日推荐后的自动风控日志
grep "PortfolioRisk" backend/logs/quant-platform.log | tail -5
# 预期：DAILY_RECOMMENDATION 后出现风控检查日志
```

## P3-10：A/B 测试框架

### 状态：❌ 未实现

### 预期目标

为推荐系统提供 A/B 测试能力：
- 同一策略同时运行多个参数版本（如 ICW 上限 30% vs 35% vs 40%）
- 按 Regime 分组对比不同因子组合的表现
- 统计显著性检验（t-test / bootstrap），确认差异非随机
- 自动淘汰显著劣势版本

### 未实现原因

优先级最低（P3），ROI 低于其他任务。当前推荐系统仍处于"验证有效性"阶段，A/B 测试需要足够的流量和样本量才有统计意义，目前 14 个策略的推荐量不足以支撑有效 A/B 测试。

### 建议实现时机

当策略数量 > 20 且每日推荐量 > 200 条时，A/B 测试框架才有足够样本量。建议在 P0-P2 全部稳定运行 3 个月后启动。

## P3-11：因子时效性管理

### 改造前

因子一旦设为 `ACTIVE` 就永久参与推荐，无衰减检测和降级机制。

**问题**：
- 市场风格切换后，原有效因子可能失效（如 SIZE 因子在 2024 年有效，2025 年失效）
- 失效因子继续参与 ICW 权重计算，稀释有效因子权重
- 无自动降级和复活机制，需要人工发现和干预
- 因子状态变更无日志记录，无法追溯

### 改造后

新增 `FactorHealthMonitor` 核心服务，实现因子全生命周期管理：

**1. 衰减检测**

| 条件 | 动作 |
|------|------|
| 90 日 IC < 启用时 IC 的 50% | 预警（`DEGRADE_WARNING`） |
| 90 日 IC < 启用时 IC 的 30% | 降级（`DEGRADED`） |
| 连续 20 日 \|IC\| < 0.015 | 降级（`DEGRADED`） |
| 连续 10 日 \|IC\| > 0.03 且 IR > 0.2 | 复活（`RESURRECTED`） |

**2. 状态联动**
- `DEGRADED` 因子自动从推荐管线过滤（`RecommendationService.getFactorConfig()` 查询时排除）
- 因子降级/复活时发布 `FactorStatusChangedEvent`（P3-12 事件总线）
- 监听器接收事件 → 自动刷新 `FactorMetaCacheService` 缓存

**3. 日志记录**
- MySQL `factor_health_log` 表：记录每次状态变更（类型/原因/时间）
- ClickHouse `factor_health_metric` 表：记录 90 日 IC、IR 等健康指标

**4. 定时调度**
- `FACTOR_HEALTH_CHECK` 任务：每日 22:00 自动执行（cron `0 0 22 * * 1-5`）

**5. REST API**

| 接口 | 功能 |
|------|------|
| `POST /api/factor-health/check` | 执行健康检查 |
| `GET /api/factor-health/report` | 查看健康报告 |
| `POST /api/factor-health/degrade/{code}` | 手动降级 |
| `POST /api/factor-health/resurrect/{code}` | 手动复活 |

### 改造价值

| 维度 | 说明 |
|------|------|
| 自动化 | 因子失效自动降级，无需人工巡检 |
| 时效性 | 每日 22:00 检查，次日推荐即过滤失效因子 |
| 可追溯 | 完整日志记录状态变更原因和时间 |
| 可恢复 | 失效因子可自动复活，避免永久淘汰有效因子 |
| 缓存一致 | 事件总线保证因子状态变更后缓存即时刷新 |

### 验证方式

```bash
# 1. 执行健康检查
curl -X POST "http://localhost:8080/api/factor-health/check"
# 预期：35 个 ACTIVE 因子检查通过

# 2. 查看健康报告
curl "http://localhost:8080/api/factor-health/report"
# 预期：每个因子的 90 日 IC、IR、状态

# 3. 手动降级 + 验证过滤
curl -X POST "http://localhost:8080/api/factor-health/degrade/MOM5"
# 预期：MOM5 status → DEGRADED，事件发布，缓存刷新

# 4. 确认 DEGRADED 因子不在推荐管线
curl -X POST "http://localhost:8080/api/recommendation/generate?dateMode=today"
# 检查日志：MOM5 不在 ICW 权重中

# 5. 手动复活
curl -X POST "http://localhost:8080/api/factor-health/resurrect/MOM5"
# 预期：MOM5 status → ACTIVE，事件发布，缓存刷新

# 6. 查看状态变更日志
SELECT * FROM factor_health_log ORDER BY created_at DESC LIMIT 10;
```

## P3-12：服务解耦重构

### 改造前

核心服务间通过直接方法调用耦合：

```
FactorComputeEngine → RecommendationService → FactorMetaCacheService
                  ↘ FactorHealthMonitor
ScheduleService → PortfolioRiskService → FactorMetaCacheService
```

**问题**：
- 6+ 条直接依赖，改一处牵动多处
- 下游异常（如缓存刷新失败）阻断上游（因子计算回滚）
- 测试需 mock 所有下游服务
- 新增消费者（如"计算完成后发通知"）需修改上游代码

### 改造后

引入 Spring `ApplicationEventPublisher` 事件总线：

**3 个新事件类型**

| 事件 | 发布者 | 携带数据 |
|------|--------|----------|
| `FactorComputeCompletedEvent` | FactorComputeEngine | factorDate, rowCount, duration |
| `RecommendationGeneratedEvent` | RecommendationService | recommendDate, strategyId, count |
| `FactorStatusChangedEvent` | FactorHealthMonitor | factorCode, oldStatus, newStatus, reason |

**1 个监听器**

| 监听器 | 监听事件 | 动作 |
|--------|----------|------|
| `FactorStatusChangedEventListener` | `FactorStatusChangedEvent` | 刷新 `FactorMetaCacheService` 缓存 |

**架构变化**：依赖关系从网状 → 星型（所有服务只依赖事件总线）

### 改造价值

| 维度 | 说明 |
|------|------|
| 可测试性 | 测试只需验证事件发布，无需 mock 5+ 下游 |
| 可扩展性 | 新增消费者 = 加监听器，上游零改动（开闭原则） |
| 容错隔离 | 下游异常不阻断上游（缓存刷新失败 ≠ 因子计算失败） |
| 可维护性 | 依赖关系从网状 → 星型，认知负载降低 |
| 演进速度 | 新功能 = 新事件 + 新监听器，并行开发不阻塞 |
| 可观测性 | 事件 = 天然审计日志，链路追踪零成本接入 |

### 验证方式

```bash
# 1. 手动降级因子，验证事件发布 → 缓存刷新链路
curl -X POST "http://localhost:8080/api/factor-health/degrade/MOM20"
# 检查日志：
grep "FactorStatusChangedEvent" backend/logs/quant-platform.log | tail -3
# 预期日志：
# "发布 FactorStatusChangedEvent: MOM20, ACTIVE → DEGRADED"
# "监听器接收 FactorStatusChangedEvent，刷新 FactorMetaCacheService 缓存"

# 2. 手动复活，验证反向链路
curl -X POST "http://localhost:8080/api/factor-health/resurrect/MOM20"
# 预期日志：
# "发布 FactorStatusChangedEvent: MOM20, DEGRADED → ACTIVE"
# "监听器接收，刷新缓存"

# 3. 因子计算完成后验证事件发布
# 触发 FACTOR_COMPUTE 任务后检查日志：
grep "FactorComputeCompletedEvent" backend/logs/quant-platform.log | tail -1
# 预期：日志包含 factorDate、rowCount、duration

# 4. 推荐生成后验证事件发布
grep "RecommendationGeneratedEvent" backend/logs/quant-platform.log | tail -1
# 预期：日志包含 recommendDate、strategyId、count
```

## 附录：技术约定速查

| 约定 | 说明 |
|------|------|
| `next_day_excess_return` 存储单位 | 百分比数值（如 -7.03 = -7.03%），显示不再 ×100 |
| ICW 无 IC 数据默认权重 | 0.05（非 1.0），防止新因子权重过高 |
| `FactorType` 枚举 | BUILTIN / SCRIPTED / COMPOSITE / PATTERN（4 种） |
| `FactorStatus` 枚举 | DRAFT / TESTING / ACTIVE / DEPRECATED / DEGRADED（5 种） |
| P1-4 噪声阈值 | \|IC\| < 0.015 直接剔除（NOISE_DROPPED），不反转 |
| P3-11 衰减阈值 | 90 日 IC < 启用时 50% → 预警；< 30% → 降级 |
| P3-11 复活条件 | 连续 10 日 \|IC\| > 0.03 且 IR > 0.2 |
| P3-11 检查时间 | 每日 22:00（cron `0 0 22 * * 1-5`） |
| P3-12 事件总线 | Spring `ApplicationEventPublisher`（未上 Kafka/RabbitMQ） |
| P1-6 并行线程数 | 5（CompletableFuture + 固定线程池） |
| P2-8 缓存 TTL | 6 小时（动态行业相关性） |
| P2-8 聚类阈值 | 0.70（Union-Find） |

| P2-9 单股仓位上限 | 15% |
| P2-9 组合回撤上限 | 20% |
