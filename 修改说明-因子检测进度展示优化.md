# 因子检测进度展示优化 & 状态同步修复

## 修改时间
2026-03-16

## 修改目的
根据用户需求对因子检测进度展示进行优化，并修复状态同步问题：
1. 删除列表上方的进度条，避免重复显示
2. 支持刷新页面后点击进行中的检测行，展示检测过程日志及进度
3. **修复**：检测完成后状态不更新，刷新页面后重复检测的问题
4. **修复**：新建检测后状态显示"待运行"（应为"运行中"）
5. **修复**：点击已完成的检测后，检测过程仍然显示的问题
6. **新增**：批量删除功能，列表中增加第一列展示ID

#### 2.2 新增handleStartMonitoring函数
**位置**: 在handleStopTest函数之后添加

**功能**: 
1. 清理旧的WebSocket连接
2. 初始化进度和日志状态
3. 建立新的WebSocket连接监听检测进度
4. 根据WebSocket消息更新检测日志（数据准备、IC检测、分组检测等）
5. 启动轮询检查检测状态
6. 检测完成后自动刷新列表

**代码逻辑**:
```javascript
const handleStartMonitoring = (report) => {
  // 1. 清理旧的WebSocket连接
  if (testStompRef.current) {
    testStompRef.current.deactivate();
  }
  
  // 2. 设置进度和日志
  setRunningReportId(report.id);
  setTestProgress(5);
  setTestLogs([]);
  pushTestLog('📌 开始监听检测进度', 'info');

  // 3. 启动WebSocket监听
  const testClient = new Client({
    brokerURL: WS_URL,
    debug: str => console.log('[TEST STOMP]', str),
    onConnect: () => {
      pushTestLog('🔌 WebSocket 已连接，开始监听检测进度', 'info');
      testClient.subscribe(`/topic/factor/${factor.factorCode}`, msg => {
        const data = JSON.parse(msg.body);
        if (data.stage?.startsWith('TEST')) {
          // 处理各种阶段的日志
          if (data.stage === 'TEST_START') {
            // 数据准备阶段
          } else if (data.stage === 'TESTING') {
            // 检测计算阶段（IC检测、分组检测等）
          } else if (data.stage === 'TEST_DONE') {
            // 检测完成
          }
        }
      });
    },
    onStompError: frame => {
      console.error('[TEST STOMP ERROR]', frame);
      pushTestLog(`❌ STOMP 错误: ${frame.headers.message}`, 'error');
    }
  });
  
  testStompRef.current = testClient;
  testClient.activate();

  // 4. 启动轮询检查状态
  if (testPollRef.current) clearInterval(testPollRef.current);
  testPollRef.current = setInterval(() => {
    setTestProgress(prev => {
      if (prev < 60) return Math.min(prev + 4, 60);
      if (prev < 80) return Math.min(prev + 2, 80);
      if (prev < 92) return Math.min(prev + 1, 92);
      return prev;
    });
    factorApi.getTestReport(report.id).then(r => {
      if (r.data.status === 'COMPLETED' || r.data.status === 'FAILED') {
        clearInterval(testPollRef.current);
        setTestProgress(100);
        setRunningReportId(null);
        testStompRef.current?.deactivate();
        loadFactor();
        if (r.data.status === 'COMPLETED') message.success('因子检测完成！');
        else message.error('检测失败: ' + r.data.errorMessage);
      }
    });
  }, 3000);
};
```

## 修复说明：检测完成后状态不同步的问题

### 问题描述
用户反馈：检测过程中日志显示已完毕，但列表中的检测行仍然是运行中，刷新页面后，又开始重新检测，出现重复的检测。

### 问题原因
在收到WebSocket的`TEST_DONE`消息时，前端只关闭了WebSocket连接，但没有：
1. 清空`runningReportId`状态
2. 设置`testProgress`为100
3. 调用`loadFactor()`刷新列表

导致：
- 列表中的检测状态仍然是"运行中"
- 刷新页面后，前端认为检测还在运行，重新启动WebSocket监听
- 但后端检测已经完成，导致显示异常

### 解决方案
修复了三处`TEST_DONE`消息的处理逻辑，在收到完成消息后立即：

```javascript
} else if (data.stage === 'TEST_DONE') {
  const match = data.message?.match(/reportId=(\d+)/);
  pushTestLogWithStage('DONE', `🎉 检测完成！报告ID: ${match?.[1] || '?'}`, 'success');
  testClient.deactivate();
  setRunningReportId(null);      // 清空运行中报告ID
  setTestProgress(100);          // 设置进度为100%
  loadFactor();                  // 立即刷新列表
}
```

### 修复的位置
1. **第281-288行**：`runTestAfterCompute`函数中的WebSocket处理
   - 场景：因子值未计算，先触发计算再运行检测

2. **第399-406行**：`handleTest`函数else分支中的WebSocket处理
   - 场景：因子值已存在，直接运行检测

3. **第545-552行**：`handleStartMonitoring`函数中的WebSocket处理
   - 场景：刷新页面后点击运行中的检测行

### 修复效果
✅ 检测完成后，列表状态立即更新为"已完成"
✅ 进度条显示为100%
✅ 立即刷新列表显示最新的检测数据
✅ 刷新页面后不会重新启动WebSocket监听
✅ 不会再出现重复检测的问题

### 备份机制
即使WebSocket的`TEST_DONE`消息处理失败，还有轮询机制作为备份：
- 每3秒轮询一次检测状态
- 当检测状态为`COMPLETED`或`FAILED`时，自动更新状态并刷新列表
- 确保状态最终能正确同步

## 新增修复：状态显示异常问题

### 问题1：新建检测后状态显示"待运行"
**用户反馈**：新建检测后，列表出现的检测行，状态显示待运行（不正确）

#### 原因分析
1. `FactorService.triggerTest()`方法创建测试报告时，使用Builder模式构建
2. 初始状态未设置，默认为`PENDING`（待运行）
3. 创建报告后立即返回给前端
4. 虽然在异步方法`runFactorTest()`中会设置状态为`RUNNING`，但由于异步执行，前端收到的是`PENDING`状态

#### 解决方案
**文件**: `backend/src/main/java/com/quant/platform/factor/service/FactorService.java`

**修改位置**: 第235-251行

```java
// 修改前
FactorTestReport report = FactorTestReport.builder()
        .factorCode(factor.getFactorCode())
        .testName(testName)
        .startDate(startDate)
        .endDate(endDate)
        .stockPool(stockPool)
        .rebalanceFreq(rebalanceFreq)
        .build();
report = testReportRepo.save(report);
```

```java
// 修改后
FactorTestReport report = FactorTestReport.builder()
        .factorCode(factor.getFactorCode())
        .testName(testName)
        .startDate(startDate)
        .endDate(endDate)
        .stockPool(stockPool)
        .rebalanceFreq(rebalanceFreq)
        .status(FactorTestReport.TestStatus.RUNNING)  // 立即设置为运行中
        .build();
report = testReportRepo.save(report);
```

**效果**：
✅ 新建检测后，列表中立即显示"运行中"状态
✅ 与异步方法`runFactorTest()`中的状态设置保持一致
✅ 用户体验更好，状态显示准确

### 问题2：点击已完成检测后，检测过程仍然显示
**用户反馈**：点击其他已完成的检测，但检测过程还是出现，应该不展示，在切换回运行中的检测行后再显示

### 用户体验优化

1. **智能按钮显示**：
   - 没有选中行时，不显示"批量删除"按钮
   - 选中行后，按钮立即显示，并显示选中的数量

2. **二次确认机制**：
   - 删除前必须确认，防止误操作
   - 包含运行中检测时，提示更加明显和详细

3. **加载状态**：
   - 批量删除时按钮显示loading状态
   - 防止重复点击

4. **清理机制**：
   - 删除后自动清空选中状态
   - 清空WebSocket连接（如果包含运行中的检测）
   - 刷新列表显示最新数据

### 功能演示

**操作步骤**：
1. 勾选要删除的检测报告（可以全选或多选）
2. 点击"批量删除 (N)"按钮
3. 根据选中的检测类型，显示对应的确认对话框
4. 确认后执行删除，显示成功提示
5. 自动刷新列表

**支持的组合**：
- ✅ 只删除已完成的检测
- ✅ 只删除运行中的检测
- ✅ 混合删除（同时包含已完成和运行中的检测）

### 注意事项

1. 点击运行中的检测行时，必须确保 `factor?.factorCode` 存在，否则无法建立WebSocket连接
2. `handleStartMonitoring` 函数会自动清理旧的WebSocket连接，避免内存泄漏
3. 进度条的最大值为100%，当检测完成时设置为100%
4. WebSocket连接失败时会在日志中显示错误信息
5. **重要**：在`TEST_DONE`消息处理中必须执行三步操作：
   - `setRunningReportId(null)` - 清空运行中状态
   - `setTestProgress(100)` - 设置完成进度
   - `loadFactor()` - 刷新列表状态
